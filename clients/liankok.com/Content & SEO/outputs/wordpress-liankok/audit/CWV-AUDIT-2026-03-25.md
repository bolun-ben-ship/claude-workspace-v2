# Core Web Vitals & Performance Audit
**Site:** https://liankok.com
**Date:** 2026-03-25
**Auditor:** RightClick:AI — Web Performance Specialist
**Method:** HTML source analysis + curl timing (PSI API keyless quota exhausted)

---

## Executive Summary

liankok.com has severe performance problems driven by three compounding factors: **51 render-blocking JavaScript files**, **91 CSS stylesheets loaded synchronously**, and **zero image optimisation** (no WebP/AVIF, no caching plugin, no CDN). The TTFB at ~1.1s is already borderline, and the JS/CSS waterfall on top of it will push LCP well into the Poor range on mobile. This site will fail all three Core Web Vitals on mobile and likely fail LCP on desktop.

## Issue Registry

### CRITICAL

#### C1 — 51 Render-Blocking JavaScript Files
**Impact: LCP, FCP, INP**

51 of 63 `<script>` tags load with no `defer` or `async` attribute. Every one of these blocks HTML parsing and delays first paint. The repairex2 theme alone contributes 22 blocking scripts:

- `jquery.min.js`, `jquery-migrate.min.js` (WordPress core — unavoidable but should be bottom of body)
- `bootstrap.min.js`, `popper.min.js` (theme)
- `bxslider.js`, `owl.js`, `isotope.js`, `parallax.min.js`, `parallax-scroll.js` (animation libraries)
- `circle-progress.js`, `countdown.js`, `tilt.jquery.js`, `wow.js` (effect libraries)
- `fancybox.js` (lightbox — loaded by both theme AND gutenkit-blocks-addon)
- Elementor core: `frontend.min.js`, `frontend-modules.min.js`, `webpack.runtime.min.js`
- ElementsKit Lite: `frontend-script.js`, `widget-scripts.js`
- WPForms Lite: `wpforms.min.js`, `wpforms-modern.min.js`, `jquery.validate.min.js` + 3 more
- `download-now-for-woocommerce/somdn_script.js`

**The theme ships 22 individual JS files with no bundling.** This is an extremely high waterfall for a commerce site.

Fix: Apply `defer` to all non-critical scripts. Use a caching/optimisation plugin (WP Rocket, LiteSpeed Cache, or Autoptimize) to concatenate and defer theme JS.

---

#### C2 — 91 CSS Stylesheets (All Render-Blocking)
**Impact: LCP, FCP**

The browser must download and parse all 91 CSS files before it can render anything. Key offenders:

- **repairex2 theme:** 22 individual CSS files (bootstrap.css, global.css, responsive.css, responsive-2.css, color.css, color-2.css, blog.css, hover.css, elpath.css, flaticon.css, flaticon-2.css, font-awesome-all.css, owl.css, nice-select.css, etc.)
- **Gutenkit Blocks Addon:** 14 separate block-level CSS files loaded globally on every page regardless of whether those blocks are used
- **Elementor:** 18+ CSS files including per-post CSS for post IDs 1291, 1243, 2002, 2258, 1685, 2831, 3139, 3192
- **ElementsKit Lite:** 3 CSS files
- **WooCommerce:** 4 CSS files
- **WPForms Lite:** 1 CSS file
- **Google Fonts:** 2 separate `fonts.googleapis.com` requests (one from theme, one from Elementor — same Inter font loaded twice)

**Google Fonts is loaded from an external domain with no preconnect hint**, adding DNS + TCP + TLS overhead to font delivery.

Fix: Critical CSS inline + defer non-critical. Remove duplicate Google Fonts calls. Add `<link rel="preconnect" href="https://fonts.googleapis.com">` and `<link rel="preconnect" href="https://fonts.gstatic.com" crossorigin>`. Disable Gutenkit CSS for unused blocks.

---

#### C3 — No Image Optimisation: 0 WebP, 0 AVIF, 213 JPG/PNG References
**Impact: LCP, bandwidth**

Every image on the site is a legacy format. Homepage has 44 `<img>` tags; /products/ has 52. Not a single image uses WebP or AVIF. On mobile networks this is a significant LCP contributor — a hero PNG at 525×525px is likely 150–400KB unoptimised.

Additional image issues:
- 6 images on homepage without `loading="lazy"` (these load eagerly and compete with the LCP element)
- 11 images on homepage without `srcset` (full-size desktop images served to mobile)
- 33 images on /products/ without `loading="lazy"` — product catalogue images all loading immediately
- No `fetchpriority="high"` on the actual LCP element (the two `fetchpriority=high` found are likely incorrect or on non-LCP elements)
- Product images named `Untitled-design-15.png`, `Untitled-design-91.png` — Canva exports served directly, not optimised

Fix: Run all uploads through ShortPixel or Imagify (WebP auto-conversion). Add `loading="lazy"` to all images below the fold. Ensure the single LCP image (hero/first visible image) has `fetchpriority="high"` and is preloaded with `<link rel="preload" as="image">`.

---

#### C4 — No Caching Plugin Detected
**Impact: TTFB, repeat visits, all CWV**

No caching plugin markers are present in the HTML. No WP Rocket, W3 Total Cache, WP Super Cache, LiteSpeed Cache, or Autoptimize. This means:

- Every page request hits PHP + MySQL with no page cache
- Static assets (JS, CSS, images) have no guaranteed long-term cache headers
- No HTML minification
- No asset concatenation
- No critical CSS extraction
- TTFB will remain ~1s+ for every visitor

Fix: Install WP Rocket (paid, recommended) or LiteSpeed Cache (free, excellent). This single action will improve TTFB, reduce blocking resources, enable lazy loading, and generate critical CSS — addressing C1, C2, C3, and C4 simultaneously.

### HIGH

#### H1 — TTFB at 1,071ms — Server Response Too Slow
**Impact: LCP subpart "TTFB", all CWV**

Google's LCP TTFB threshold is 800ms for "good". At 1.07s the server is already over budget before any assets load. Root cause is likely no page caching (see C4) combined with a shared hosting environment. The SSL handshake (35ms) and DNS (2ms) are fast — the bottleneck is PHP/MySQL execution time.

Fix: Install a page caching plugin (primary fix). If TTFB remains above 800ms after caching, investigate hosting upgrade to a VPS or managed WordPress host (Kinsta, WP Engine, Cloudways).

---

#### H2 — No LCP Image Preload
**Impact: LCP**

There are zero `<link rel="preload" as="image">` tags in the `<head>`. The LCP element (most likely a hero image or product banner) will only start downloading after the browser has parsed enough of the HTML to discover it. On a page with 91 blocking CSS files, this discovery is severely delayed.

Fix: Identify the LCP element (hero image URL), add `<link rel="preload" as="image" href="..." fetchpriority="high">` in the `<head>`. This is the single highest-impact LCP fix after caching.

---

#### H3 — Heavy INP Risk: Elementor + ElementsKit + Gutenkit + WooCommerce + WPForms on Main Thread
**Impact: INP**

Four overlapping page builder / block addons are active simultaneously:
- **Elementor** (core + Pro)
- **ElementsKit Lite** (Elementor addon)
- **Gutenkit Blocks Addon** (Gutenberg addon)

This is an unusual combination — Elementor and Gutenberg blocks are typically used exclusively, not together. Running both means double the JS runtime, double the event listeners, and more main-thread contention on interaction. Combined with WooCommerce and WPForms JS, the main thread will be heavily occupied during page load, which directly increases INP.

Fix: Decide whether the site uses Elementor or Gutenberg (Gutenkit) and remove the unused stack. Disable ElementsKit widgets not actively used on pages.

---

#### H4 — Duplicate Fancybox Library (Theme + Gutenkit)
**Impact: INP, unused JS**

`jquery.fancybox.js` is loaded by both the repairex2 theme and separately by gutenkit-blocks-addon (`fancybox.js`). This is ~50KB of duplicate lightbox JS executing twice, registering duplicate event listeners.

Fix: Disable the fancybox enqueue from whichever plugin/theme is not the source of truth for lightbox usage. Use a plugin like Asset CleanUp Pro to conditionally dequeue by page.

### MEDIUM

#### M1 — Google Fonts Loaded Twice (Inter font, two separate requests)
**Impact: LCP, FCP, font flash**

Two separate `fonts.googleapis.com` requests for the Inter font family — one from the repairex2 theme and one from Elementor. This doubles the external font latency and can cause FOUT (Flash of Unstyled Text) on both load paths.

Fix: Remove the duplicate. Ideally self-host Inter using Google Fonts Helper and serve from the same origin to eliminate the external DNS lookup entirely.

---

#### M2 — No Google Fonts Preconnect
**Impact: FCP, LCP**

No `<link rel="preconnect" href="https://fonts.googleapis.com">` or `<link rel="preconnect" href="https://fonts.gstatic.com" crossorigin>` in the `<head>`. Without preconnect, the browser must complete DNS + TCP + TLS for Google's font servers before font CSS can download.

Fix: Add both preconnect hints as the very first `<link>` tags in `<head>`.

---

#### M3 — 57.5 KB of Inline CSS Across 42 Style Blocks
**Impact: HTML parse time, caching**

42 separate `<style>` blocks totalling 57.5 KB of inline CSS are embedded in the HTML. Inline CSS cannot be cached separately and bloats the HTML document. Most of this comes from Elementor's per-element inline styles. The 301 KB HTML payload is very large for a homepage.

Fix: Enable Elementor's "Improved Asset Loading" experiment (Elementor > Settings > Experiments) to move inline CSS to external files. Combined with a caching plugin, this reduces HTML size substantially.

---

#### M4 — CLS Risk: 6 Images Without Lazy + Missing Dimensions on 1 Image
**Impact: CLS**

One image is missing both width and height attributes. When such images load they cause layout shifts. Additionally, 6 above-the-fold images loading without lazy could cause reflow if their containing elements have no reserved space.

The second `<img>` in the body has `src=""` — an empty src attribute which triggers a spurious browser request and may cause a layout shift when the real image loads.

Fix: Audit and add explicit `width` and `height` to all images. Remove or fix the `src=""` empty image tag. Reserve space for dynamic content (sliders, carousels) with aspect-ratio CSS.

---

#### M5 — 22 Separate Theme JS Files, No Concatenation
**Impact: HTTP requests, LCP, INP**

The repairex2 theme loads 22 individual JavaScript files. Even with HTTP/2 multiplexing, each file has overhead: header parsing, browser processing, and separate execution contexts. Many of these are specialised animation libraries (wow.js, parallax, tilt, countdown) that may only be used on specific pages or sections.

Fix: Use WP Rocket or Autoptimize to concatenate theme JS into a single file. Audit with Asset CleanUp Pro to disable specific scripts on pages where they are not used.

---

#### M6 — Font Awesome Loaded in Full (Theme + Elementor)
**Impact: CSS size, LCP**

The repairex2 theme loads `font-awesome-all.css` (the complete Font Awesome library, ~70KB). Elementor also loads its own Font Awesome CSS (`all.min.css`, `solid.min.css`, `brands.min.css`, `regular.min.css`, `fontawesome.min.css`). This is approximately 200–300KB of icon CSS, most of it unused on any given page.

Fix: Enable Font Awesome subsetting in Elementor settings. Consider replacing icon fonts with SVG icons for critical UI elements.

---

### LOW

#### L1 — WPForms Lite JS Loaded on All Pages
**Impact: unnecessary JS on non-form pages**

WPForms Lite loads 5 JS files (`jquery.validate.min.js`, `mailcheck.min.js`, `punycode.min.js`, `utils.min.js`, `wpforms.min.js`) on every page, including pages with no forms. This is ~30–40KB of unnecessary JS increasing main thread work on product pages.

Fix: Use WPForms' "Load assets only on pages with forms" setting (available in WPForms > Settings > General) or use Asset CleanUp Pro to restrict WPForms scripts to form pages only.

---

#### L2 — Facebook Pixel Loaded Synchronously via noscript Tag
**Impact: minor, tracking accuracy**

A Facebook Pixel noscript `<img>` tag is the very first element in the `<body>`. While noscript tags don't block rendering, this is an indicator the pixel implementation may not be using the standard async script loader correctly.

Fix: Verify the Facebook Pixel is implemented via the standard `<script>` async snippet and not relying on the noscript fallback as the primary tracking method.

---

#### L3 — Product Images Named After Canva Exports
**Impact: SEO, alt text, accessibility**

Images named `Untitled-design-15.png` and `Untitled-design-91.png` are Canva exports served directly without renaming. This provides zero SEO value to filenames. Alt attributes on product images appear to be empty strings.

Fix: Rename all uploaded images with descriptive, keyword-rich filenames before upload (e.g., `abb-mcb-circuit-breaker-sh200l.png`). Add descriptive alt text to all product images.

## WordPress/Elementor-Specific Notes

- **Theme: repairex2** — a multi-purpose repair/service theme repurposed for an electrical distributor. Not purpose-built for e-commerce. Ships an excessive number of JS/CSS files for animation effects that may not be used (parallax, countdown timers, text animation, tilt effects). Consider a lighter WooCommerce-native theme (Astra, GeneratePress, Kadence) long-term.
- **Elementor + Elementor Pro** — running both is correct, no issue there. However Elementor's "Improved Asset Loading" and "Inline Font Icons" experiments should be enabled in Elementor > Settings > Experiments to reduce inline CSS and icon library overhead.
- **ElementsKit Lite** — an Elementor addon adding its own scripts and styles. Audit which widgets are actually in use and disable the rest via ElementsKit dashboard.
- **Gutenkit Blocks Addon** — loads 14 block-specific CSS files globally regardless of page content. This should only be active if Gutenberg blocks are being used alongside Elementor. If all pages are built with Elementor, this plugin should be deactivated.
- **Google Site Kit** — not detected in asset loading but listed as active. Ensure it is not loading additional tracking scripts synchronously.
